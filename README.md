# README — запуск сайта на GitHub Pages + полная настройка

Этот проект — статический сайт с:
- интерфейсом bio-страницы,
- встроенной админкой (редактирование контента прямо на сайте),
- паролем на админку,
- статистикой заходов и внешних кликов,
- облачной синхронизацией через Supabase (чтобы данные были общими для всех пользователей, а не только в одном браузере).

## Что в проекте

Файлы:
- `index.html` — структура страницы и админки.
- `styles.css` — стили сайта.
- `main.js` — вся логика: рендер, админка, пароль, статистика, cloud sync.
- `SUPABASE_SETUP.md` — SQL и базовая инструкция по Supabase.

## Быстрый старт локально

### Вариант A (самый простой)
1. Открой `index.html` в браузере.
2. Сайт уже работает в локальном режиме (данные в `localStorage`).

### Вариант B (рекомендуется для cloud-режима)
Запусти локальный HTTP-сервер (чтобы не было ограничений `file://` в некоторых браузерах):
- через VS Code extension **Live Server**, или
- через любой статический сервер.

После запуска открой локальный URL (например `http://127.0.0.1:5500`).

## Публикация на GitHub Pages

### 1) Создай репозиторий
1. На GitHub создай новый репозиторий (например `my-links-site`).
2. Загрузи в корень репозитория файлы:
   - `index.html`
   - `styles.css`
   - `main.js`
   - `README.md`
   - `SUPABASE_SETUP.md`

### 2) Залей код
Если через Git командой:

```bash
git init
git add .
git commit -m "initial site"
git branch -M main
git remote add origin https://github.com/<USERNAME>/<REPO>.git
git push -u origin main
```

### 3) Включи GitHub Pages
1. Открой репозиторий на GitHub.
2. Перейди: **Settings → Pages**.
3. В блоке **Build and deployment** выбери:
   - **Source**: `Deploy from a branch`
   - **Branch**: `main`
   - **Folder**: `/ (root)`
4. Сохрани настройки.

### 4) Получи ссылку на сайт
Через 1–3 минуты появится URL вида:
- `https://<USERNAME>.github.io/<REPO>/`

Если репозиторий называется `<USERNAME>.github.io`, тогда URL обычно:
- `https://<USERNAME>.github.io/`

## Подключение Supabase (обязательно для общей админки/статистики)

> Без Supabase данные редактируются только локально в браузере.

### 1) Создай проект в Supabase
1. Создай новый проект.
2. Открой **Project Settings → API**.
3. Скопируй:
   - `Project URL`
   - `anon public key`

### 2) Создай таблицы и политики
Открой SQL Editor в Supabase и выполни SQL из файла `SUPABASE_SETUP.md`.

Это создаёт:
- `site_content` — хранение контента сайта (1 запись по ключу, например `main`),
- `site_events` — события статистики (`visit`, `external_click`).

### 3) Проверь RLS
Убедись, что:
- RLS включен для обеих таблиц,
- политики созданы,
- `anon` имеет разрешения на select/insert/update/delete согласно SQL.

## Настройка админки на сайте

### 1) Открой сайт
Открой GitHub Pages URL в браузере.

### 2) Вход в админку
1. Нажми `open admin`.
2. Если пароль ещё не создан:
   - введи новый пароль (минимум 4 символа),
   - нажми `unlock`.
3. Если пароль уже есть — введи его и нажми `unlock`.

### 3) Включи cloud-режим
В блоке cloud-полей в админке:
- включи `Cloud mode (Supabase)`,
- `Supabase URL` = URL проекта,
- `Supabase anon key` = anon key,
- `Content table` = `site_content`,
- `Events table` = `site_events`,
- `Content row key` = `main`.

### 4) Первая синхронизация
1. Нажми `save changes` (сохранение локально + попытка cloud push).
2. Нажми `cloud push` (принудительно залить текущий контент в облако).
3. Нажми `cloud sync stats` (подтянуть облачную статистику).

### 5) Фото в стиле карточки (как на скрине)
В админке доступны поля:
- `Profile photo upload` — загрузка фото,
- `Profile photo URL` — ссылка на фото,
- `Profile photo title` — большой текст внизу,
- `Profile photo badge` — символ бейджа (например `✔`),
- `Profile photo link` — куда ведёт карточка.

После изменений нажми `save changes`.

### 6) Управление доступом
- `lock admin` — заблокировать админку сразу.
- `change password` — сменить пароль (спросит текущий и новый).

## Как обновлять сайт после публикации

Любые изменения в файлах (`index.html`, `styles.css`, `main.js`) публикуются так:
1. Измени файлы локально.
2. Commit + push в ветку `main`.
3. Подожди 1–3 минуты.
4. Обнови страницу сайта (`Ctrl+F5` для принудительного обновления).

## Чеклист проверки после деплоя

Проверь по порядку:
1. Сайт открывается по GitHub Pages URL.
2. `open admin` открывает панель.
3. Пароль работает (unlock / lock / change password).
4. Изменение текста и фото сохраняется после перезагрузки.
5. В cloud-режиме:
   - `cloud push` проходит без ошибок,
   - `cloud pull` подтягивает актуальный контент,
   - `cloud sync stats` показывает общие данные.
6. Клик по внешним ссылкам увеличивает `external clicks`.
7. Повторный заход увеличивает `visits`.

## Частые проблемы и решения

### 1) `cloud pull failed` / `cloud push failed`
Причины:
- неверный `Supabase URL`,
- неверный `anon key`,
- таблицы/политики не созданы,
- RLS не настроен.

Что сделать:
- перепроверь поля в админке,
- заново выполни SQL из `SUPABASE_SETUP.md`,
- проверь имена таблиц (`site_content`, `site_events`).

### 2) Статистика не меняется
- Нажми `cloud sync stats`.
- Убедись, что клики идут именно по внешним `http/https` ссылкам.
- Проверь, что запись в `site_events` разрешена для `anon`.

### 3) После изменения файлов сайт не обновился
- Подожди 1–3 минуты (Pages деплой не мгновенный).
- Нажми `Ctrl+F5`.
- Проверь, что изменения действительно в ветке `main`.

### 4) Забыл пароль админки
Пароль хранится в `localStorage` текущего браузера.
Варианты:
- открыть сайт в другом браузере и задать новый пароль там,
- очистить `localStorage` для домена сайта в DevTools.

## Безопасность: что важно понимать

Сейчас проект настроен как удобный и быстрый MVP для GitHub Pages:
- используется публичный `anon key` на фронте,
- админ-пароль хранится клиентски (в браузере),
- при открытых RLS-политиках запись в таблицы доступна по ключу.

Для прод-уровня защиты рекомендуется:
- добавить серверную авторизацию,
- ужесточить RLS (запись только для авторизованного пользователя),
- вынести критичные операции на backend/edge functions.
